## 사장님 (가게 운영자)

> "감이 아니라 숫자로 가게를 운영하세요"

### 가치 흐름

```
1. 트렌드 파악   →  주/월별 별점 변화 추이 확인
       ↓
2. 문제 특정     →  카테고리별 분석으로 정확한 문제 영역 식별
       ↓
3. 조치 실행     →  데이터에 기반한 구체적 개선 실행
       ↓
4. 효과 측정     →  다음 주기 분석에서 개선 효과 확인
```

### 핵심 기능

| 기능 | 사장님 입장에서의 가치 |
|---|---|
| 별점 트렌드 차트 | 매출 그래프처럼 직관적. 평점 변화를 1초에 파악 |
| 카테고리별 별점 | "음식은 좋은데 배달이 문제" → 즉각 대응 가능 |
| 전주/전월 비교 | 포장 교체 후 배달 점수 변화 → 효과 측정 |
| AI 요약 리포트 | 100개 리뷰를 3줄로 요약. 핵심 불만/칭찬 즉시 파악 |
| 급변 알림 | "위생 불만 300% 증가" 같은 위기 신호 조기 감지 |
| 거래량(리뷰 수) | 리뷰 적은 달은 신뢰도 낮음 인지 → 과민반응 방지 |

---

## 소비자 (리뷰 소비자)

> "100개 리뷰 안 읽고 핵심만 파악하세요"

### 가치 흐름

```
1. 트렌드 확인   →  최근 별점 추이로 가게 상태 파악
       ↓
2. 카테고리 필터 →  음식/배달/서비스 등 관심 분야만 확인
       ↓
3. AI 요약 확인  →  AI 생성 리포트로 핵심 내용 파악
       ↓
4. 주문 결정     →  데이터 기반 신뢰할 수 있는 선택
```

### 핵심 기능

| 기능 | 소비자 입장에서의 가치 |
|---|---|
| 별점 트렌드 차트 | "4.0점이어도 하락 추세면 경고" |
| 카테고리별 별점 | "배달 빠른 곳" 등 중요 항목만 확인 |
| AI 요약 | 100개 리뷰 안 읽고 "맛있고 양 적다"를 1초에 파악 |
| 월별 추이 | "예전엔 좋았는데 요즘 별로"인 가게 회피 |
| 키워드 빈도 | 다수 vs 소수 의견 구분. 3/100명 불만이면 무시 가능 |

## 요약 vs 통계 — 둘 다 필요한 이유

| | 통계만 | 요약만 | ReviewTime (둘 다) |
|---|---|---|---|
| 문제 파악 | "3.1점, 하락" | "배달 불만 있음" | "배달 3.1점(-0.5), 비 오는 날 온도 불만 집중" |
| 조치 가능성 | 낮음 | 중간 | **높음** (원인 + 심각도) |
| 추적 가능성 | 높음 | 낮음 | **높음** (숫자 비교 + 맥락) |

## 종합 가치 제안

ReviewTime은 하나의 플랫폼으로 두 가지 문제를 동시에 해결합니다.

```
kiwipiepy 로컬 분석    → 비용 $0, 75~85% 자동 처리
AI 활용 요약          → 토큰 25배 절약, 정확도 유지
주차별 트렌드 시각화  → 직관적인 데이터 기반 판단
```

> **사장님의 의사결정과 소비자의 주문 판단,**
> **모두를 위한 데이터 기반 리뷰 인사이트 플랫폼**
